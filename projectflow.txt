1. Setup the virtual environment
    >> run `python -m venv .venv`
    >> run `.\.venv\Scripts\Activate.ps1`

2. Creating the project template
    >> run `python template.py`.

3. Install required packages and modules
    >> run `pip install -r requirements.txt`
    >> verify installation `pip list`

4. Coding sequence should be:

    1. Build `data_loader.py`
        >> Goal:
            >> Load all .md and .json files recursively from your data/ directory and inspect them.
        >> Your immediate objectives:
            >> Traverse folders recursively
            >> Read markdown files
            >> Read JSON files
            >> Store them in a structured format
            >> Print loaded outputs for debugging
        >> After loading:
            >> print total documents
            >> print file paths
            >> print sample contents

    2. Build chunking.py
        >> Goal:
            >> Split large documents into smaller retrievable chunks.
        >> You now need to learn:
            >> why chunking matters
            >> chunk size effects
            >> overlap effects
            >> semantic continuity
        >> Print chunks and visually inspect:
            >> where splits occur
            >> whether sentences break badly
            >> whether important context gets lost
    This stage strongly affects retrieval quality later.

    3. Build embedding.py:
        >> Goal:
            >> Convert chunks into numerical semantic vectors using the OpenAI Embeddings API.
        >> You will now learn:
            >> what embeddings actually look like
            >> vector dimensions
            >> semantic similarity behavior
            >> embedding cost considerations
            >> batching concepts
        >> For your current phase:
            >> generate embeddings manually
            >> inspect vector outputs
            >> print dimensions
            >> store vectors alongside metadata
        >> Very important debugging checks:
            >> embedding dimension consistency
            >> empty chunk handling
            >> API failures
            >> rate limits
            >> duplicate chunks
        >> You should print:
            >> total embeddings created
            >> vector length
            >> sample vector values
            >> corresponding chunk text
    This is where your RAG system becomes semantic rather than keyword-based.

    4. vector_store.
    >> Goal:
        >> Store embeddings inside a FAISS vector index and learn how semantic retrieval actually works.
    >> You will now learn:
        >> vector indexing
        >> similarity search
        >> nearest-neighbor retrieval
        >> embedding dimensions
        >> vector ↔ metadata mapping
    >> Your flow becomes:
        Chunks
            ↓
        Embeddings
            ↓
        FAISS Index
            ↓
        Semantic Search
    >> Important concept: FAISS stores ONLY vectors.
    >> It does NOT store:
        >> chunk text
        >> file source
        >> metadata
    >> So you must separately maintain:
        >> chunk metadata list
        >> vector position mapping
    >> You should inspect:
        >> total vectors indexed
        >> embedding dimensions
        >> search distances
        >> retrieved vector IDs
    >> Very important debugging practice:
    >> After storing vectors:
        >> immediately perform a test similarity search
        >> inspect whether retrieved chunks make semantic sense
    >> That is the first real proof your RAG pipeline works.

    5. Build retriever.py:
        >> Goal:
        Given a user query:
            convert query → embedding
            serach FAISS index
            retrieve most semantically similar chunks

        You will now finally observe:
            semantic search behavior
            relevance ranking
            retrieval failures
            chunk quality effects

        Flow:
            User Query
            ↓
            Query Embedding
            ↓
            FAISS Similarity Search
            ↓
            Top Matching Chunks

        Very important debugging outputs:
            query text
            query embedding dimension
            similarity distances
            retrieved chunk IDs
            retrieved sources
            retrieved chunk text

    6. Build prompt_builder.py:
        Goal:
            Construct the final prompt sent to the LLM using:
                retrieved chunks
                user query

        This is where retrieval becomes actual RAG.

        Flow becomes:
            User Query
            ↓
            Retrieve Relevant Chunks
            ↓
            Inject Context Into Prompt
            ↓
            Send To LLM

        You will now learn:
            grounding
            hallucination reduction
            context injection
            prompt engineering
            token

        Most Important Learning Objective

        Print the FULL final prompt before sending it to OpenAI.

        Inspect:

            context quality
            redundancy
            chunk ordering
            prompt length
            hallucination risk

        This stage teaches one of the most important RAG skills:

            retrieval-to-generation alignment

        A huge number of RAG systems fail here even when retrieval works correctly.
            
    7. Build generator.py:
        Goal:
            Send the final RAG prompt to the OpenAI chat model and generate the final answer.

        Now your complete RAG pipeline becomes:
            Documents
            ↓
            Chunking
            ↓
            Embeddings
            ↓
            FAISS Retrieval
            ↓
            Prompt Construction
            ↓
            LLM Generation

        This is the first fully working end-to-end RAG system.
            What You Will Learn Here
            grounding vs hallucination
            prompt effectiveness
            retrieval-generation alignment
            context limitations
            answer quality evaluation
            temperature 
            
        Very Important Debugging Practice

            Print:

                retrieved chunks
                final prompt
                final answer

            before optimizing anything.

        You want to observe:

            when retrieval succeeds
            when generation ignores context
            when hallucinations happen
            when prompt instructions fail

This develops true RAG engineering intuition.

    8. main.py
        Goal:
            Orchestrate the complete RAG pipeline cleanly.

        You now already have:

            loader
            chunker
            mbeddings
            vector store
            reteriever
            prompt builder
            generator

        main.py becomes the central execution entrypoint.

        our Initial Pipeline Flow
            Load Documents
            ↓
            Chunk Documents
            ↓
            Create Embeddings
            ↓
            Build FAISS Index
            ↓
            Save Vector Store
            ↓
            Interactive Query Loop
            ↓
            Retrieve Chunks
            ↓
            Build Prompt
            ↓
            Generate Answer
        
        Important Architectural Insight

            Separate your system into TWO phases:

                1. Indexing Phase
                Runs occasionally.
                documents
                → chunks
                → embeddings
                → FAISS
                → save index

                This is expensive.

                2. Query Phase
                Runs repeatedly.
                query
                → retrieval
                → prompt
                → generation

                This should be fast.

            Very Important Learning Point
                You should NOT regenerate embeddings every time a user asks a question.
                Many beginners accidentally do this.
                Embedding generation belongs to indexing pipeline not query pipeline
    
    At every step:
        print intermediate outputs
        inspect structures
        inspect dimensions
        inspect chunk boundaries
        inspect retrieval scores
        inspect prompts
